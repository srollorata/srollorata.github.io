## What I do

Evaluate and improve code design using SOLID principles and creative patterns:

### SOLID Principles

**S - Single Responsibility**
- Ensure classes have one reason to change
- Separate concerns appropriately
- Avoid god objects

**O - Open/Closed**
- Design for extension, not modification
- Use abstractions to accommodate changes
- Implement plugin architectures where appropriate

**L - Liskov Substitution**
- Ensure subclass contracts match parent
- Maintain behavioral compatibility
- Avoid breaking existing interfaces

**I - Interface Segregation**
- Prefer small, focused interfaces
- Avoid monolithic interfaces
- Client-specific abstractions

**D - Dependency Inversion**
- Depend on abstractions, not concrete implementations
- Use dependency injection
- Reduce coupling through interfaces

### Creative Design

- Propose innovative solutions beyond textbook patterns
- Adapt patterns to specific project needs
- Consider trade-offs between simplicity and flexibility
- Balance YAGNI with future-proofing
- Explore alternative approaches that may better fit context

## When to use me

Use this when:
- Designing new components or modules
- Refactoring existing code
- Reviewing architectural decisions
- Solving complex design problems
- Balancing competing requirements (simplicity, performance, maintainability)

Ask clarifying questions about project constraints, existing patterns, and specific requirements to tailor recommendations.
